## 1. rohitg00/agentmemory — コーディングエージェントに「記憶」を持たせる共通基盤

- リポジトリ: [https://github.com/rohitg00/agentmemory](https://github.com/rohitg00/agentmemory)

### どんなツールか

agentmemoryは、Claude CodeやCursor、Codex、Gemini CLIといった複数のコーディングエージェントが共有して使える永続メモリのレイヤだ。エージェントがセッションをまたぐたびにアーキテクチャの説明や好みのライブラリを毎回教え直す、というあの面倒な体験を、エージェント側の実装を変えずに解消することを狙っている。MCP・REST・各エージェントのフックという3系統のインタフェースを揃えていて、特定のフレームワークに縛られない点が特徴だ。

### 主な特徴

- セッション開始時にBM25とベクトル検索のハイブリッド検索で関連メモリを取得し、設定したトークン予算（デフォルト2000トークン）に収まる範囲で文脈として注入する
- PostToolUseなど12種のフックでエージェントの操作を自動キャプチャし、SHA-256による重複排除とシークレットの除去を経て構造化された事実として保存する
- メモリをworking／episodic／semantic／proceduralの4階層に整理し、エビングハウスの忘却曲線に近い形で減衰・強化される仕組みを備える
- SQLiteベースで外部DB不要。Qdrantやpgvectorのセットアップが要らないので、個人開発でもすぐ導入できる
- 複数エージェントが同じメモリを参照する前提で設計されており、リースやシグナルといった協調プリミティブを持つ

### なぜいま注目されているのか

エージェント向けメモリの分野はmem0やLetta、supermemoryなど競合がすでに賑わっているが、agentmemoryが伸びている理由は明確で、「自動キャプチャ」と「マルチエージェント共有」の両立にある。mem0系は基本的にAPI経由で能動的に書き込む設計なので、エージェント側のプロンプトや実装に手を入れないと記憶が増えていかない。agentmemoryはフック経由で勝手にメモリが溜まるので、ユーザは普段どおりエージェントを使うだけでよい。

さらに、ここ数か月でClaude CodeやCodex、Cursorなどを併用する開発者が一気に増えた結果、「Cursorで決めた設計をClaude Codeにも引き継ぎたい」というニーズが顕在化している。MCP越しに同じメモリプールを参照できるagentmemoryは、この乗り換え・併用シナリオに素直に答える形になっており、トレンド入りも頷ける。トークン消費を年間65万から17万程度に抑えられるという主張も、エージェント運用コストに悩んでいる開発者にとっては実利が大きい。

## 2. millionco/react-doctor — エージェントが書く「微妙なReact」を機械的に検出する

- リポジトリ: [https://github.com/millionco/react-doctor](https://github.com/millionco/react-doctor)

### どんなツールか

react-doctorは、Reactコードベースを走査して0〜100のヘルススコアと診断結果を出すCLIツールだ。リポジトリの一文目に「Your agent writes bad React. This catches it.」と書かれている通り、AIエージェントが生成したReactコードに頻出するアンチパターンを拾い上げることに振り切っている。Next.js、Vite、React Nativeに対応し、検出ルールはフレームワークやReactのバージョンに応じて自動的に切り替わる。

### 主な特徴

- ステート管理、エフェクト、パフォーマンス、アーキテクチャ、セキュリティ、アクセシビリティの各カテゴリに分かれた静的解析ルール群を備える
- スコア算出は `100 - (ユニークなerrorルール数 × 1.5) - (ユニークなwarningルール数 × 0.75)` という単純な式で、違反箇所の総数ではなくルールの種類数で測るため、機械的なコピー&ペーストでスコアが大きく下振れしない
- `--diff`モードで指定ブランチからの差分だけを検査でき、PR単位での退行防止に使える
- GitHub Actions向けのcomposite actionが用意されており、PRコメントとしてスティッキーに結果を貼り付けられる
- `npx react-doctor install`を実行すると、Claude CodeやCursor、Codex、OpenCode、Windsurfなど50以上のエージェント向けに `SKILL.md` / `AGENTS.md` / `.cursorrules` といったルールファイルを生成し、生成時点でベストプラクティスを意識させられる

### なぜいま注目されているのか

エージェントが書くReactコードの典型的な問題、たとえば `useEffect` への過剰依存、派生stateの不適切な管理、メモ化の濫用や逆に必要な箇所での未使用、`use client` の意図しない伝播といった「動くけれど将来必ず痛い目を見るコード」は、レビュアー側がよほど慣れていないと見落としやすい。react-doctorはそこを「スコア」という単純な指標に落とし込んでくれるため、AI生成PRに対して人間がチェックすべき観点を絞り込める。

特筆すべきは、検査ツールであると同時に**エージェントを教育するツールとしても機能する**点だ。`install`コマンドで生成されるルールファイルをエージェントに読み込ませると、エラーを検出してから直すのではなく、最初からアンチパターンを避けたコードが出やすくなる。検出→修正のループを回すツールは多いが、生成前にルールを注入できる設計は近年広がっているSkills/AGENTS.md系のエコシステムとも自然に噛み合う。

エージェントによるコード生成が日常化した結果、「CI上で機械的に弾けるAIコードの品質基準が欲しい」という需要が高まっている。react-doctorはまさにそのニッチに刺さっており、Reactに領域を絞っていることが逆に「鋭い検出ルール」を実装できる強みになっている。同種のツールがVueやSvelte向けにも広がっていく流れの先駆けとして、追っておく価値がある。
