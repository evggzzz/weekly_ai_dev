## Understand-Anything — コードベースを「触れる地図」に変えるAIプラグイン

- リポジトリ: https://github.com/Lum1104/Understand-Anything
- 公式サイト: https://understand-anything.com
- デモ: https://understand-anything.com/demo/

### どんなものか

Understand-Anythingは、巨大なコードベースやドキュメント、ナレッジベースを、クリックして探索できるインタラクティブな知識グラフに変換してくれるツールです。ソースコードを解析して、ファイル・関数・クラス・依存関係をノードとして可視化し、「このコードがどことつながっているのか」を地図のように俯瞰できるようにします。

面白いのは、Tree-sitterによる静的解析とLLMを組み合わせたハイブリッド構成になっている点です。構造そのものはパーサーが正確に拾い、その上で「この関数は業務的に何をやっているのか」「このモジュールはどのビジネスプロセスに対応するのか」といった意味づけをLLMが補ってくれます。単なる依存グラフ生成ツールではなく、コードと業務ロジックの橋渡しまでやってくれるのが他のツールと違うところです。

### 実際に何がうれしいのか

一番効くのは、新しいプロジェクトにジョインしたときや、長年放置されたレガシーコードに手を入れるときです。誰も全体像を把握していない数十万行のコードに対して、いきなり「どこから読めばいいか」をガイド付きツアーで提示してくれます。しかもツアーは依存関係の順序で並んでいるので、土台になる部分から積み上げる形で理解を進められます。

レビューの場面でも刺さります。Diff Impact Analysisを使うと、ある変更がどこまで波及するかをグラフ上で視覚的に確認できるので、「この修正、本当にここだけで済む?」という不安に対して具体的な答えをくれます。あいまい検索とセマンティック検索の両方に対応しているため、関数名をうろ覚えでも「だいたいこういう処理」というニュアンスで目的の箇所にたどり着けるのも実用的です。生成されるドキュメントは日本語を含む複数言語に対応しているので、海外製OSSの理解にも使えます。

### なぜ今これが効くのか

エージェントにコードを書かせるのが当たり前になった結果、逆に「人間が全体を把握しきれていないコード」が増えるという皮肉な状況が生まれています。生成は速いけれど、それを後から読み解いて安全に直すコストはむしろ上がっている。Understand-Anythingが今これだけ伸びているのは、その「生成と理解のギャップ」を埋めるツールへの需要が一気に高まっているからだと感じます。

しかもこのツールはClaude Code、Cursor、VS Code + Copilot、Copilot CLI、Codex、Gemini CLIなど主要なAIコーディング環境にプラグインとして横断的に対応しています。特定のツールに縛られず、自分が普段使っている環境にそのまま組み込めるのは、乗り換えや並行利用が当たり前になっている今の開発スタイルとよく噛み合っています。

## microsoft/agent-governance-toolkit — エージェントの暴走を「構造的に不可能」にする統制レイヤー

- リポジトリ: https://github.com/microsoft/agent-governance-toolkit
- ドキュメント: https://microsoft.github.io/agent-governance-toolkit

### どんなものか

Agent Governance Toolkit（AGT）は、自律エージェントに対する本番運用向けのコントロールシステムです。マイクロソフトが開発しているもので、エージェントが実行するすべてのツール呼び出し・メッセージ・委譲を、実行される前にアプリケーション層で横取りし、ポリシーに従って許可・拒否・承認待ちに振り分けます。

設計思想がはっきりしていて、READMEの一節が端的です。いわく「プロンプトレベルの安全策は、確率的なシステムに対する丁寧なお願いにすぎない」。つまり「危ないことはやらないでね」とプロンプトで頼むのではなく、エンフォースメントをモデルの外側にある決定的なミドルウェアへ移す。これによってポリシー違反を「起きにくくする」のではなく「構造的に起こせなくする」というアプローチを取っています。

### 実際に何がうれしいのか

使い方はシンプルで、既存のツール関数を `govern()` でラップするだけです。`safe_tool = govern(my_tool, policy="policy.yaml")` のように包めば、あとはYAMLで書いたルールに従って制御が効きます。ポリシーは allow / deny / require-approval の3アクションで記述でき、たとえばデータベースのdropやdeleteのような破壊的操作は人間のレビューなしで問答無用にブロックし、メール送信やDB書き込みのような繊細な操作は承認フローを挟む、といった制御が宣言的に書けます。

うれしいのは、特定のエージェントフレームワークに縛られないことです。AutoGen、LangGraph、CrewAI、Claude Code、OpenAI Agents SDKなど、手元のスタックが何であっても上からかぶせられます。さらにゼロトラストのアイデンティティ（SPIFFE / DID / mTLS）、4段階の権限リングによる実行サンドボックス、改ざん検知付きの監査ログ、MCP向けのセキュリティゲートウェイ（ツールポイズニング検知やタイポスクワッティング対策）まで揃っていて、OWASPのAgentic Top 10で挙げられているリスクを一通りカバーしている点も心強いところです。監査ログはOWASPやNIST、EU AI Act、SOC 2といったコンプライアンス要件へのマッピングも意識されています。

### なぜ今これが効くのか

ここ最近、エージェントを「実験」から「本番」へ移そうとして詰まるケースが急増しています。デモではうまく動くのに、いざ実際の権限を渡そうとすると「もしこいつが本番DBを消したら誰が責任を取るのか」という問いが立ちはだかる。プロンプトのお願いベースでは監査にもセキュリティレビューにも通らない、というのが多くのチームの本音だと思います。

AGTが今トレンド入りしているのは、まさにこの「本番投入の最後の壁」に対する具体的な答えを提示しているからです。マイクロソフトが手がけているという信頼感に加え、Python / TypeScript / .NET / Rust / Go と多言語SDKが揃い、formal specに対する1,000近いconformanceテストで挙動を担保しているあたりからも、遊びではなく本気で本番運用を見据えていることが伝わってきます。エージェントを「動かす」フェーズから「安全に任せる」フェーズへ移りつつある今、押さえておきたい一本です。
